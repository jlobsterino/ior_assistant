## ЧАСТЬ 1. Сводка изменений по аналитическим гипотезам и выгрузкам (из CHANGES.md и CHANGESV2.md)

### 1. Добавленные файлы

#### 📄 `backend/agent/hypothesis.py`
* **Роль**: Движок оффлайн-гипотез (`HypothesisEngine`) и генератор графиков.
* **Функционал**:
  * **Расчет статистического профиля датафрейма**: включает автоматическое выявление концентрации рисков по **Правилу Парето** (долю потерь топ-1% и топ-5% крупнейших инцидентов) и расчет статистических выбросов по методу $3\sigma$ (среднее плюс три стандартных отклонения).
  * **Генерация двух типов графиков**:
    1. **Динамика потерь и инцидентов во времени** (`generate_dynamics_chart`): столбчатая диаграмма количества инцидентов (изумрудный цвет `#10b981`) + линия суммы потерь (розовый `#f43f5e`). Если потери отсутствуют или равны нулю, шкала потерь и красная линия автоматически скрываются, предотвращая плоский график на нуле. Временной шаг адаптируется под размер выборки (до 31 дня — по дням с хронологической сортировкой `ДД.ММ`, до 180 — по неделям, иначе — по месяцам).
    2. **Распределение по категориям** (`generate_distribution_chart`): автоматически сканирует срезы данных (ТБ, тип события, статус или авторегистрация). Если категорий $\le 5$, строит круговую диаграмму долей (Donut Chart); если больше (от 6 до 12) — строит горизонтальный график столбцов.
  * **Премиальный темный дизайн**: Отрисовка графиков переведена на темно-синюю подложку (`#0b0f19`), которая идеально сливается с темным фоном личного кабинета приложения.
  * **Асинхронная генерация**: Картинки строятся в отдельных потоках через `asyncio.to_thread` для предотвращения блокировок event loop бэкенда на больших выборках.
* **Ссылка на файл**: [hypothesis.py](file:///d:/ior_assistant/backend/agent/hypothesis.py)

#### 📄 `scripts/build_local_catalog.py`
* **Роль**: Скрипт сборки оффлайн-каталогов заземления.
* **Функционал**: Генерирует файлы `kb_value_catalog.json` и `kb_value_index.json` в папке `backend/agent/schema/` на основе локальной базы DuckDB для обеспечения оффлайн-работы поиска с опечатками.
* **Ссылка на файл**: [build_local_catalog.py](file:///d:/ior_assistant/scripts/build_local_catalog.py)

Внедрено принудительное расширение списка запрашиваемых полей стандартными аудиторскими колонками (ТБ, процессы, статусы, авторегистрация, суммы потерь) для детальных выгрузок. ### 2. Защита от падений KeyError в тулах (`_resolve_df`)
При вызове трансформаций данных модель часто ошибалась в аргументе `df_id` (передавая `None` или имя таблицы). В файл `backend/agent/tools/dataframe_ops.py` добавлена функция `_resolve_df(ctx, df_id)`:
1. Если `df_id` пуст, берется единственный зарегистрированный датафрейм в сессии.
2. Если `df_id` является именем таблицы, она автоматически загружается из хранилища (DuckDB/Spark) и регистрируется на лету.
3. Этот метод интегрирован во все основные инструменты трансформации (`filter_df`, `group_by`, `derive_column`, `window_rank`, `export_excel`, `join_dfs`).

## ЧАСТЬ 4. Дополнительные исправления интерфейса, SSE-стриминга, FAISS-сопоставления и предустановленных скриптов (Июль 2026)

### 1. Рендеринг заголовков Markdown на фронтенде
* **Проблема**: Заголовки (H1, H2, H3 вида `# `, `## `, `### `), списки и прочее форматирование выводились на фронтенде в виде сырого текста из-за слишком упрощенного Markdown-парсера в `renderMarkdown()`.
* **Решение**: В файле [messages.jsx](file:///d:/ior_assistant/frontend/messages.jsx) доработана функция `renderMarkdown(text)`:
  * Добавлен парсинг заголовков `# `, `## `, `### ` и возвращение соответствующих элементов `<h1>`, `<h2>`, `<h3>` с красивыми отступами и цветом заголовков.
  * Добавлена поддержка списков на основе символов `* `, `• ` и `- `.

### 2. Исправление бесконечной прокрутки лоадера в SSE-стриминге
* **Проблема**: В цепочке шагов (status timeline) этап «Анализирую данные и генерирую гипотезу...» оставался в состоянии вращения (spinning/`active`) даже после полного завершения генерации отчета.
* **Решение**: В файле [agent_flow.py](file:///d:/ior_assistant/backend/agent/agent_flow.py) сразу после завершения вызова асинхронной функции `generate_hypothesis_narrative` отправляется статусное SSE-событие `"done"` для шага `"hypothesis"`, завершающее лоадер на интерфейсе.

### 3. Исправление отображения «0 строк» при повторном открытии сессии
* **Проблема**: При формировании выгрузки количество строк показывается верно, но при перезагрузке страницы или открытии диалога из истории количество строк под кнопкой скачивания сбрасывалось в `0 строк`.
* **Решение**:
  * В файле [agent_flow.py](file:///d:/ior_assistant/backend/agent/agent_flow.py) метаданные файла выгрузки (имя, размер, количество строк, структура колонок и превью) теперь сохраняются внутри поля `"excel"` метаданных сообщения в БД (`MessageRepo.add`).
  * Аналогичные метаданные `"excel"` теперь сохраняются при прямом поиске в файле [chat.py](file:///d:/ior_assistant/backend/api/routes/chat.py).
  * При восстановлении истории сессии фронтенд ([ior-app.jsx](file:///d:/ior_assistant/frontend/ior-app.jsx)) считывает эти сохраненные данные, предотвращая сброс счетчика строк в `0`.

### 4. Исправление сопоставления FAISS для символьных ID (incdnt_sid)
* **Проблема**: При последующих уточняющих вопросах в диалоге агент не мог сопоставить выбранные из выгрузки строковые идентификаторы вида `EVE-XXXXXXX` с FAISS-индексом, выводя в лог ошибку сопоставления.
* **Решение**: В файле [IOR_pipeline_search.py](file:///d:/ior_assistant/backend/IOR_pipeline_search.py) в функцию `build_and_cache_small_index` внедрен глобальный поиск по хэш-карте `sid_to_index` (сопоставление SID инцидентов с их номерами строк в матрице эмбеддингов). Это восстановило контекстный диалог по выгруженным строкам.
